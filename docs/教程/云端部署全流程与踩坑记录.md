## 二、SSH 密钥生成（第一个坑）

### 错误写法
```bash
ssh-keygen -t ed25519 -C "id_autologin" -f "~/.ssh/id_autologin"
```
**坑点**：`-f` 参数用了双引号，导致 `~` **不会被 shell 展开**，最终在当前目录生成了一个真的叫 `~/.ssh/id_autologin` 的文件（字面量）。

### 正确写法
```bash
# 去掉引号，让 shell 自动展开 ~
ssh-keygen -t ed25519 -C "id_autologin" -f ~/.ssh/id_autologin

# 或者使用 $HOME 变量
ssh-keygen -t ed25519 -C "id_autologin" -f "$HOME/.ssh/id_autologin"
```

### 后续步骤
1. 把公钥 `id_autologin.pub` 内容添加到服务器的 `~ubuntu/.ssh/authorized_keys` 中
2. 确保私钥权限正确：`chmod 600 ~/.ssh/id_autologin`

## 三、deploy.sh 的进化历程

### 第一版：scp 全量删除 + 全量上传
```bash
ssh ... "rm -rf /home/ubuntu/ai-sandbox/*"
scp -i "$SSH_KEY" -r ./* ubuntu@162.14.79.120:/home/ubuntu/ai-sandbox/
```
**问题**：
- `./*` **不会复制隐藏文件**（如 `.gitignore`）
- 会连同 `node_modules/` 和 `.git/` 一起上传，又慢又脏
- 先清空后上传，如果 `scp` 中断，云端就只剩半个项目

### 第二版：改用 rsync
**问题**：Termux 上没有预装 `rsync`，尝试 `pkg install rsync` 时遇到 OpenSSL 依赖冲突，安装失败。

### 最终版：本地构建 + tar 打包 dist + Nginx 静态托管
核心思路：
1. **本地构建**（Termux 有 node，服务器上没有 node）
2. 只上传 `dist/` 目录到服务器
3. 服务器 Nginx 直接 serve 静态文件

```bash
#!/bin/bash
set -e

SERVER_IP="162.14.79.120"
PORT="4000"
SERVER_USER="ubuntu"
NAME="ai-sandbox"
SSH_KEY="$HOME/.ssh/id_autologin"
REMOTE_DIR="/home/ubuntu/$NAME"

echo "=== $NAME 部署脚本 ==="

# 1. 本地构建
echo "[1/3] 本地构建..."
npm run build

# 2. 上传 dist 到服务器
echo "[2/3] 上传 dist 到服务器..."
tar czf - dist/ | ssh -i "$SSH_KEY" "$SERVER_USER@$SERVER_IP" "cd $REMOTE_DIR && rm -rf dist && tar xzf -"

# 3. 重启 Nginx
echo "[3/3] 重启 Nginx..."
ssh -i "$SSH_KEY" "$SERVER_USER@$SERVER_IP" "sudo systemctl restart nginx"

echo ""
echo "=== 部署完成！访问 http://$SERVER_IP:$PORT ==="
```

**优点**：
- 不需要在服务器上安装 node/npm
- 不上传 `node_modules`、`.git`、日志等垃圾文件
- 构建产物小，上传快

## 四、Nginx 同时运行多个站点

### 核心机制
Nginx 通过 `sites-available/`（配置文件仓库）+ `sites-enabled/`（激活链接）来管理多站点。

### star-fighter 原有配置
```bash
/etc/nginx/sites-available/star-fighter
/etc/nginx/sites-enabled/star-fighter -> ../sites-available/star-fighter
```
监听 `8080`，root 指向 `/home/ubuntu/star-fighter`。

### 为 ai-sandbox 新增配置
在服务器上执行：
```bash
sudo tee /etc/nginx/sites-available/ai-sandbox <<'EOF'
server {
    listen 4000;
    server_name _;

    root /home/ubuntu/ai-sandbox/dist;
    index index.html;

    # HTML/JS/CSS 永不缓存
    location ~* \.(html|js|css)$ {
        add_header Cache-Control "no-cache, no-store, must-revalidate";
        add_header Pragma "no-cache";
        expires 0;
    }

    # 图片等资源长期缓存
    location ~* \.(jpg|jpeg|png|gif|ico|svg|woff|woff2)$ {
        expires 1y;
        add_header Cache-Control "public, immutable";
    }

    location / {
        try_files $uri $uri/ =404;
    }
}
EOF

# 激活配置
sudo ln -sf /etc/nginx/sites-available/ai-sandbox /etc/nginx/sites-enabled/ai-sandbox

# 检查语法并重载
sudo nginx -t
sudo systemctl reload nginx
```

**关键点**：
- 不同 `server` 块监听不同端口即可共存
- `root` 一定要指向 `dist/`（构建产物目录），不是项目根目录
- 每次部署后 `systemctl restart nginx` 或 `reload` 使其生效

## 五、防火墙大坑：ERR_CONNECTION_RESET

### 现象
部署完成后，浏览器访问 `http://162.14.79.120:4000/` 报错：
```
net::ERR_CONNECTION_RESET
```

### 排查过程
1. Nginx 已监听 4000：`ss -tlnp | grep 4000` ✅
2. Nginx 配置语法正常：`nginx -t` ✅
3. 查看 UFW 防火墙状态：
   ```bash
   sudo ufw status
   ```
   输出显示：
   - `8080 ALLOW Anywhere` ✅
   - **`4000` 没有任何规则** ❌

### 根因
`ERR_CONNECTION_RESET` 的罪魁祸首就是 **UFW 没有放行 4000 端口**。TCP 三次握手直接被防火墙拒绝并重置。

### 解决
```bash
sudo ufw allow 4000/tcp
```

放行后刷新页面即可正常访问。

## 六、完整检查清单

每次新增端口/新站点时，按这个清单检查：

- [ ] SSH 密钥已生成且已添加到服务器 `authorized_keys`
- [ ] 本地 `npm run build` 成功生成 `dist/`
- [ ] `dist/` 已上传到服务器正确位置
- [ ] Nginx `sites-available/` 配置文件已创建
- [ ] `sites-enabled/` 软链接已建立
- [ ] `sudo nginx -t` 语法检查通过
- [ ] Nginx 已 reload/restart
- [ ] **UFW 已放行对应端口**
- [ ] 浏览器能正常访问

## 七、常用命令速查

```bash
# 查看 Nginx 监听端口
ss -tlnp | grep nginx

# 查看 UFW 状态
sudo ufw status

# 放行端口
sudo ufw allow 4000/tcp

# Nginx 语法检查
sudo nginx -t

# 重载/重启 Nginx
sudo systemctl reload nginx
sudo systemctl restart nginx

# 查看 Nginx 错误日志
sudo tail -f /var/log/nginx/error.log
```

## 八、后续维护注意

- 如果以后 `ai-sandbox` 加了新依赖或改动了构建流程，确保 `vite.config.js` 里的 `outDir: 'dist'` 不变，否则 `deploy.sh` 里的路径要同步改
- 如果要换端口，需要同时修改：
  1. `deploy.sh` 里的 `PORT`
  2. Nginx `sites-available/ai-sandbox` 里的 `listen`
  3. UFW 防火墙放行新端口
