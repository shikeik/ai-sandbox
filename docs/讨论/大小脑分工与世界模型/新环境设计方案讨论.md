# 新环境设计方案讨论

> 时间：2026-04-06
> 背景：用户提出要给AI大脑提供发挥空间，需要重构游戏环境

---

## 当前环境的局限性

### 现有机制（grid-world）
- **层数**: 3层（天空、中层、地面）
- **视野**: 5x3，狐狸在x0y1（左边缘）
- **移动**: 只能向右（走/跳/远跳）
- **生死**: 即死机制（掉坑、碰敌人）
- **元素**: 空气、平地、史莱姆、恶魔、金币

### 为什么不需要大脑
现在的环境是"信息完备 + 即时反馈"的典型：
- 狐狸只需要看右边一格有没有坑
- 有坑就跳，没坑就走
- 不需要记忆（上一秒的事不影响下一秒）
- 不需要规划（走到哪算哪，没有回头路）

这就像让一个人走在直线上，只关心下一步会不会踩空——不需要大脑皮层，脊髓反射就够了。

---

## 新环境设计需求

### 核心变化
1. **血量系统** → 非即死，有持续状态
2. **血包** → 需要寻找资源，有目标导向
3. **11x5 视野居中** → 感知周围空间，不只是前方
4. **自由左右移动** → 有选择，可以回头
5. **上下平台** → 三维空间导航，不只是横移

### 为什么这些变化催生"大脑"需求

| 新机制 | 产生的问题 | 需要的认知能力 |
|--------|-----------|--------------|
| 血量 | 当前受伤会影响未来决策 | 状态记忆、风险评估 |
| 血包 | 需要找路去血包位置 | 目标导向、路径规划 |
| 左右移动 | 每一步都有"往左还是往右"的选择 | 前瞻、比较不同路径 |
| 上下平台 | 跳跃不再是固定的，要看高度差 | 空间推理、动作序列规划 |

**例子场景**：
> 狐狸在平台上，左边是血包（需要跳下去再爬上来），右边是通关点但路上有3个敌人。
> 当前血量30%，碰一个敌人扣20%。
> 
> 小脑反应：右边敌人→跳过去躲开（但跳下去上不来）
> 大脑规划：如果先去左边拿血包，回来再走右边，可以承受一次碰撞

这就是**需要世界模型**的场景——必须在想象中推演"如果我跳下去会发生什么"。

## 方案对比

### 方案1：改造 fox-jump

**现状**: fox-jump 是一个完整的横版游戏，有：
- CSS绘制的狐狸动画
- 32格地形生成
- AI控制器（NeuralNetwork.ts）
- 相机滚动、粒子效果等

**改造点**:
1. 把地形从1D（平地/坑）改为2D（支持平台高度）
2. 修改狐狸控制器支持上下移动
3. 增加血量UI和血包元素
4. 修改AI输入（从4维扩展到11x5视野）

**优点**:
- 视觉效果成熟，狐狸动画精美
- 有现成的AI训练框架

**缺点**:
- fox-jump 代码耦合度高（CSS动画、DOM操作、游戏逻辑混在一起）
- 2D平台碰撞检测复杂（需要重写JumpGame.ts的物理）
- 改动量巨大，等于重写半个游戏

---

### 方案2：升级 grid-world 系统

**现状**: terrain-lab/grid-world/ 已有：
- GridWorld.ts: 核心世界模型
- GridWorldRenderer.ts: Canvas渲染
- GridWorldAnimator.ts: 动画系统
- 完整的元素定义系统（ElementDef）

**改造点**:
1. **高度扩展**: 从3层改为5层（或更多）
2. **移动系统重构**: 
   - 支持左右移动（不只是右）
   - 支持上下移动（爬平台/跳下）
   - 重新设计ActionType
3. **新元素**: 血包、不同高度的平台
4. **状态系统**: 血量、最大血量
5. **视野扩展**: 11x5，AI居中
6. **物理重写**: 不再是"跳跃=横移"，而是真正的重力+平台碰撞

**优点**:
- 纯代码式，结构清晰，易于调试
- 与地形实验室的其他功能（训练、预测）天然兼容
- 改动模块化（改ElementDef、改Action检查、改渲染）
- 可以快速验证AI行为（不需要等动画）

**缺点**:
- 视觉效果不如fox-jump精美（emoji风格）
- 需要重新设计所有的动作逻辑

---

## 推荐方案：方案2（升级 grid-world）

理由：
1. **架构更清晰**: grid-world 本身就是为"格子世界模拟"设计的，扩展5层+左右移动是自然的
2. **AI开发友好**: 纯代码式意味着可以快速测试AI决策，不需要等CSS动画
3. **与大脑系统兼容**: 世界模型需要想象"下一帧状态"，grid-world的数据结构（二维数组）非常适合这个
4. **渐进式开发**: 可以一步步加功能（先左右移动→再上下→再血量）

### Phase 2: 物理系统重写（2-3天）

**核心问题**: 从"跳跃=横移"改为"跳跃=垂直运动+重力"

```typescript
interface HeroState {
    col: number        // x位置
    row: number        // y位置（可以是浮点数，表示空中）
    vx: number         // 水平速度
    vy: number         // 垂直速度
    onGround: boolean  // 是否在地面上
}

// 动作检查不再是"检查目标格"，而是"模拟物理结果"
checkAction(action: ActionType): {
    ok: boolean
    predictedState: HeroState  // 预测执行动作后的状态
    risks: Risk[]              // 预测的风险（如"可能掉坑"）
}
```

### Phase 3: 血量与游戏机制（1-2天）

```typescript
interface GameState {
    hero: HeroState
    hp: number           // 当前血量
    maxHp: number        // 最大血量
    enemiesKilled: number
    itemsCollected: number
    isDead: boolean
    isGoalReached: boolean
}

// 碰撞不再是即死，而是扣血
handleCollision(enemy: ElementDef): void {
    this.state.hp -= enemy.damage
    if (this.state.hp <= 0) {
        this.state.isDead = true
    }
}
```

### Phase 4: 大脑系统接入（3-5天）

**世界模型训练**:
```typescript
// 收集训练数据
interface Transition {
    state: WorldState      // 当前状态（11x5视野 + 自身状态）
    action: ActionType     // 执行的动作
    nextState: WorldState  // 下一帧状态

// 世界模型
class WorldModel {
    predict(state: WorldState, action: ActionType): {

        reward: number
        confidence: number
    }
}
```

**想象规划**:
```typescript
function planAction(currentState: WorldState, depth: number): ActionType {
    // 对每个可能动作，想象执行后的结果
    const candidates = ["左移", "右移", "上移", "下移", "跳跃"].map(action => {
        const imagined = imagineTrajectory(currentState, action, depth)
        return { action, score: evaluateTrajectory(imagined) }
    })
    
    return candidates.sort((a, b) => b.score - a.score)[0].action
}
```

## 总结

**核心洞察**: 
环境复杂度决定了需要什么样的AI。当前环境太简单，小脑就够用了。要引出"大脑"，必须先让环境复杂到"小脑不够用"。

**推荐路径**:
1. **方案2**（升级grid-world）是更务实的选择
2. 新建"大脑挑战"页面，不破坏现有功能
3. 分4个阶段逐步实现，每个阶段都可以独立测试
4. 最终目标是看到"小脑冲动地跳下去，大脑规划地绕路"的对比
