# d9 从0到1：让 AI 自己长出属性标注

> 核心问题：当前原型中 `walkable`、`agent.pos.x`、`vector` 都是人类预编码的先验。真正的元认知 AI 必须能从纯粹的感官像素中自己发现这些概念。

---

## 1. 为什么当前原型是"假的"

在之前的验证脚本中：
- AI 知道 `"上"` 对应 `vector=[0,-1]`
- AI 知道 `"wall"` 对应 `walkable=false`
- AI 知道移动前要检查"目标方向上的格子"

这些不是 AI "学"到的，是人类硬编码在 `getDirectionDelta` 和 `tileType !== "wall"` 里的。**AI 只是在执行人类已经写好的因果模型。**

真正的从0到1，应该是：
> AI 看到的世界只有 `true`/`false` 像素。它不知道哪个方向是哪个，不知道 `true` 意味着什么，不知道动作和结果之间该看哪个格子。一切都必须从实验数据中"长"出来。

---

## 2. 极简试验场：1×4 走廊

为了控制变量，我们设计一个最简环境：

```
地图：1 行 × 4 列
(0) #  (1) @  (2) .  (3) #
      ↑
    agent
```

**世界规则**：
- 每个格子只有一个二元属性：`true`（显示为 `.`）或 `false`（显示为 `#`）
- agent 只能感知**左右紧邻格子**的属性
- 动作只有两个：`左`、`右`

**实验过程**：

| 步骤 | 动作 | 左邻居 | 右邻居 | 结果 | agent 位置 |
|------|------|--------|--------|------|-----------|
| 1 | 左 | false | true | 失败 | (1) |
| 2 | 右 | false | true | 成功 | (2) |
| 3 | 右 | true | false | 失败 | (2) |
| 4 | 左 | true | false | 成功 | (1) |

---

## 3. AI 的内心独白（从0到1）

### 步骤 1 后：
> 我执行了 "左"，我没动。当时左邻居是 false，右邻居是 true。为什么没动？可能是左邻居的问题，也可能是右邻居的问题，也可能是我本来就不该动。

**生成的假设**：
- H1: "左" 的结果取决于左邻居。false → 失败
- H2: "左" 的结果取决于右邻居。true → 失败
- H3: "左" 总是失败

### 步骤 2 后：
> 我执行了 "右"，我动了。当时左邻居是 false，右邻居是 true。

**检验假设**：
- H2 说："左"的结果取决于右邻居，右邻居=true → 失败。但步骤1中右邻居确实是 true，结果是失败，H2 暂时没被推翻。
- H3 说："左"总是失败。步骤1确实失败了，暂时成立。

### 步骤 3 后：
> 我执行了 "右"，我没动。当时左邻居是 true，右邻居是 false。

**对比步骤 2 和步骤 3**：
- 动作相同（"右"）
- 结果不同（成功 vs 失败）
- 唯一显著变化：右邻居从 true 变成了 false
- **新假设**："右" 的结果取决于右邻居。true → 成功，false → 失败

### 步骤 4 后：
> 我执行了 "左"，我动了。当时左邻居是 true，右邻居是 false。

**对比步骤 1 和步骤 4**：
- 动作相同（"左"）
- 结果不同（失败 vs 成功）
- 唯一显著变化：左邻居从 false 变成了 true
- **新假设**："左" 的结果取决于左邻居。true → 成功，false → 失败

**同时检验旧假设**：
- H2（"左"取决于右邻居）：步骤1右邻居=true→失败，步骤4右邻居=false→成功。右邻居变化和结果变化呈反比，H2 被推翻！
- H3（"左"总是失败）：步骤4成功了，H3 被推翻！

### 顿悟时刻：
> 哦！原来 "左" 和 "右" 是两种**不同**的动作。"左" 看左边的邻居，"右" 看右边的邻居。而邻居的属性如果是 true，我就能移动过去；如果是 false，我就过不去。

**元规则诞生**：
```typescript
{
  action: "$direction",
  condition: "$direction 的邻居 = true",
  effect: "移动成功"
}
```

## 4. 这就是从0到1的突破

在这个过程里，AI 自己"长"出了三个概念：

### 概念 1：方向绑定
AI 发现 `"左"` 和 `"右"` 不是抽象的符号，而是各自对应一个**固定的空间关系**：
- `"左"` ↔ 左边的邻居
- `"右"` ↔ 右边的邻居

人类没有告诉它这一点，它是通过**控制变量法**（对比相同动作在不同前置状态下的结果）自己推导出来的。

### 概念 2：可通行性（walkable）
AI 发现邻居属性的 `true`/`false` 和移动结果有稳定的对应关系：
- `true` → 成功
- `false` → 失败

它不需要知道这个属性叫 `walkable`。它可以在内部称之为 `feature_0_success_predictor`。

### 概念 3：移动的元规则
AI 发现这不是两条孤立的规则（左规则和右规则），而是一条泛化的元规则：
> 对于任何方向，如果该方向上的邻居是 true，就能朝那个方向移动。

这意味着：如果以后加入 `"上"`、`"下"` 动作，AI 只需要 1-2 次实验就能猜出它们也遵循同样的模式。

---

## 5. 工程实现路径

要实现上述能力，需要三个全新模块：

### 模块 A：`MinimalWorld`（极简世界）
- 每个格子只有 `true`/`false`
- agent 只能看到相邻格子的属性
- 动作结果完全由相邻格子的属性决定
- **没有任何人类预编码的语义**（没有 `walkable`，没有 `wall`）

### 模块 B：`HypothesisGenerator`（假设生成器）
- 对于每个 `(action, observation, result)`，生成所有可能的解释假设
- 假设的形式：`IF [某个邻居的属性 = X] THEN [结果是 Y]`
- 不预设哪个邻居和动作相关，而是**枚举所有可能性**

### 模块 C：`ContradictionFilter`（矛盾排除器）
- 当新的实验数据与某个假设矛盾时，删除该假设
- 保留那些从未被证伪的假设
- 当只剩下一个假设时，AI 就"确信"了这条规则

### 模块 D：`Meta-Inductor 2.0`
- 比较多个动作的 surviving hypotheses
- 发现它们的结构相似性（都是"动作绑定到特定邻居"+"邻居属性决定结果"）
- 泛化为元规则

## 6. 与当前原型的关键差异

| 维度 | 当前原型（假） | 目标原型（真） |
|------|---------------|---------------|
| 动作含义 | 人类预编码 `vector=[0,-1]` | AI 自己发现 `"上"` 绑定到上方邻居 |
| 属性含义 | 人类预编码 `walkable=true` | AI 自己发现 `true` 与成功相关 |
| 相关格子 | 人类硬编码"看目标方向上的格子" | AI 通过枚举假设+排除矛盾找到相关格子 |
| 规则形式 | 结构化对象（人类设计的数据结构） | 假设-验证-幸存者模式 |
