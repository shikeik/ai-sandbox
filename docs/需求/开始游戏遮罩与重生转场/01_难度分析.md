## 现状考察

### 代码结构概览


### 关键发现

#### 1. 状态管理现状（**中等复杂度**）

```javascript
// JumpGame.js 当前状态
export const STATUS = {
  IDLE: 'idle',      // 待机，可接受操作
  MOVING: 'moving',  // 移动中
  DEAD: 'dead',      // 死亡
  WON: 'won'         // 胜利
}
```

**问题**：
- 没有 "未开始" 状态，游戏加载即处于可玩状态
- 计时器在 `onGenerationChange` 中启动，而非"开始游戏"动作

#### 2. 重生流程现状（**需要大改**）

```javascript
// JumpGame.js 第 291-317 行
triggerDeath() {
  // ...
  setTimeout(() => this._nextGeneration(), 1500)  // 直接重生
}

triggerWin() {
  // ...
  setTimeout(() => this._nextGeneration(), 1500)  // 直接重生
}

_nextGeneration() {
  this.generation++
  this.init()  // 立即重置世界
  // ...
}
```

**问题**：
- 死亡/胜利后直接跳变到新世界
- 没有中间转场层可以介入
- `init()` 立即重置状态，无法做渐暗效果

#### 3. 渲染层现状（**较成熟，可复用**）

```javascript
// GameRenderer.js
class Tween {
  // 已有成熟的补间动画系统
  // 支持 onComplete 回调
}
```

**优势**：
- 补间动画系统成熟，可复用做渐暗渐亮
- `showDeath()` / `showWin()` 已有视觉反馈

#### 4. 计时器现状（**需调整**）

```javascript
// main.js
game.onGenerationChange = (gen) => {
  // ...
  // 玩家模式下启动计时
  if (!isAIMode) {
    game.startTimer()  // 目前这里启动
  }
}
```

**问题**：
- 计时器在世代变化时启动，而非"开始游戏"动作
- 需要改为：点击"开始游戏"按钮后才启动

## 难度评估

### 综合难度：**中等偏高** ⭐⭐⭐⭐☆

主要难点不在功能本身，而在于**需要理顺现有混乱的代码逻辑**。

## 详细风险分析

### 🔴 高风险点

#### 1. 状态机重构（影响面广）

| 现状 | 需求 | 风险 |
|------|------|------|
| 5 个状态 | 需增加 READY 状态 | 所有状态判断需检查 |
| 加载即可玩 | 需等待"开始" | 事件绑定时机需调整 |

**涉及修改**：
- `JumpGame.js` - 新增 READY 状态常量
- `main.js` - 初始化后进入 READY 而非直接可玩
- 所有键盘/按钮事件 - 增加 READY 状态检查

#### 2. 重生时序重组（逻辑复杂）

当前流程：
```
death → setTimeout(1500ms) → _nextGeneration() → init() → 新世界
```

新需求流程：
```
death → 等待(800ms) → 渐暗(400ms) → 重生 → 渐亮(400ms) → 新世界
                              ↑
                         在这里执行 _nextGeneration() + init()
```

**难点**：
- 需要拆分 `_nextGeneration()` 的执行时机
- 渐暗期间游戏状态不应变化，但视觉上要淡出
- 需要新的 `TRANSITIONING` 状态防止输入

#### 3. 计时器逻辑变更（容易遗漏）

当前：
- 世代变化 → 自动启动计时

新需求：
- 点击"开始游戏" → 启动计时
- 重生转场完成后 → 重置并启动计时

**容易遗漏的场景**：
- AI 模式 vs 玩家模式的计时区别
- 刷新页面后的计时器状态
- 从 AI 模式切换到玩家模式后的计时器

---

### 🟡 中风险点

#### 4. 遮罩层与现有层级的冲突

现有层级：
- z-index 1000: game-info
- z-index 2000: 菜单按钮
- z-index 2001: 菜单下拉

**遮罩层需要**：
- z-index 1500: 开始遮罩（高于游戏，低于菜单）
- z-index 1600: 转场遮罩（同上）

#### 5. CSS 动画与 JS 动画的协调

渐暗渐亮可以用 CSS transition，但时机控制需要 JS。

```javascript
// 可能的问题：CSS transition 与 JS 回调不同步
overlay.style.opacity = '1'  // 开始渐暗
setTimeout(() => {
  // 假设 400ms 后完成，但实际 CSS 也是 400ms
  // 如果不同步会有闪烁
}, 400)
```

#### 6. 热更新兼容性

Vite HMR 会重新执行模块，遮罩状态可能丢失。

---

### 🟢 低风险点

#### 7. 样式实现

CSS 遮罩层简单：
```css
.game-overlay {
  position: absolute;
  inset: 0;
  background: rgba(0,0,0,0.7);
  display: flex;
  align-items: center;
  justify-content: center;
  z-index: 1500;
}
```

#### 8. 按钮样式

复用现有按钮样式变量即可。

## 建议

### 立即执行
1. 先完成阶段一（代码清理），这会大幅降低后续难度
2. 从"开始游戏遮罩"入手，逻辑相对独立

### 需要特别小心
- `_nextGeneration()` 的调用时机 - 这是核心重生逻辑
- 计时器启动点 - 多处依赖，容易遗漏
- 状态机扩展 - 检查所有 switch/case 和 if 判断

### 是否需要进一步拆分
如果担心风险过高，可以：
1. **先做阶段一（代码清理）** - 降低维护成本
2. **只做"开始游戏遮罩"** - 相对独立，风险低
3. **重生转场延后** - 需要更多设计
