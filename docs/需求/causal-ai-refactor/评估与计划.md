# causal-ai 代码质量评估与重构计划

> 分支: `causal-ai-refactor`  
> 基线: `causal-ai` (d8 合并后)  
> 评估日期: 2026-04-13

---

## 一、总体概况

| 指标 | 现状 | 评估 |
|---|---|---|
| 总代码量 | ~3,329 行 | 中等规模 |
| 最大单文件 | `renderer.ts` 689 行 | 🔴 严重超标（>100 必须拆分） |
| 次大文件 | `game-controller.ts` 367 行 | 🟠 偏大 |
| 测试覆盖 | 仅有调试脚本 `test/ts/causal-ai/debug.ts` | 🔴 无正式单元测试 |
| 类型安全 | 多处 `as` 断言、字符串索引调用 | 🟡 中等风险 |

---

## 二、问题详细分析

### 🔴 1. DRY — 重复代码（严重）

| 重复点 | 分布文件 | 说明 |
|---|---|---|
| 移动动作位置计算 | `planner.ts` (`applyRule` + `canApply`)、`game-controller.ts` (`checkRuleApplicable`) | 三处重复解析 `at(agent,x,y)`、计算目标坐标、检查 `cell_empty(tx,ty)` |
| 状态转谓词调用 | `executor.ts`、`command-executor.ts`、`game-controller.ts` | 反复手写 `stateToPredicates(agent.pos, agent.facing, agent.inventory.includes("钥匙"), view)` |
| 移动方向映射 | `world.ts` (`handleMove`)、`planner.ts` (`applyRule`) | 方向到 `dx,dy` 的 `switch` 重复 |
| DOM 元素获取 | `main.ts` 散落多处 | 直接 `document.getElementById("cmdInput")` 等，未收口 |

### 🔴 2. SRP — 职责混乱（严重）

| 文件 | 问题 |
|---|---|
| `renderer.ts` (689行) | 一肩挑：CSS 注入、DOM 结构生成、图块渲染、玩家实体管理、相机跟随系统、动画控制、层级管理。典型的"上帝类"。 |
| `game-controller.ts` | `planTo()` 长达 80+ 行，把**规划逻辑**和**调试 UI 输出**死死绑在一起。无法单独测试规划，也无法单独测试输出格式。 |
| `main.ts` | 同时负责全屏/横屏 API 封装、初始化流程、事件绑定、指令执行封装、Tab 切换。 |

### 🟠 3. 架构/设计缺陷（中高）

| 问题 | 影响 |
|---|---|
| `main.ts` 中用 `controller["render"]?.()` 字符串索引调用私有方法 | 破坏封装，TypeScript 编译后可能失效，极其脆弱 |
| `CommandContext.plannedActions` 是公开数组，被 `command-executor.ts` 直接 `length = 0` + `push(...)` | 外部直接修改 GameController 内部状态，无封装 |
| `LocalView` 类型被 `getGlobalView()` 返回 | 命名撒谎，语义混乱 |
| `game-controller.ts` 中 `explore()` 直接调用 `Math.random()` | 不可测试的副作用 |

### 🟡 4. 魔法数字与硬编码

| 位置 | 问题 |
|---|---|
| `planner.ts` | `maxDepth: number = 100` 无具名常量 |
| `command-executor.ts` | 规划深度再次硬编码 `100` |
| `game-controller.ts` | `explore(count = 10)`、`sleep(100)` 无配置 |
| `world.ts` | `getLocalView(range = 3)` 实际产生 7×7 视野，但 CLI 提示"5×5"，不一致 |
| `renderer.ts` | 动画时长、相机过渡曲线等仍散落 |

### 🟡 5. DOM 操作不集中

| 位置 | 问题 |
|---|---|
| `main.ts` 重置按钮 | 直接 `cmdInput.value = ""`、`viewBtn.textContent = "..."` |
| `main.ts` 视野切换按钮 | 直接 `document.getElementById("viewToggleBtn")` |
| `renderer.ts` | `innerHTML` 注入大量内联 CSS 样式，与样式文件职责混合 |

### 🟡 6. 类型安全与命名

| 问题 | 说明 |
|---|---|
| `Action` 与 `ActionType` 并存 | 同一语义在 `core/world/types.ts` 和 `web/types.ts` 中重复定义 |
| `game-controller.ts` | `as import("./types").AgentState` 这种内联 import 类型非常怪异 |
| `renderer.ts` | `obj.state?.open as boolean \| undefined` 不必要的类型断言 |
| 日志标签 | `[CAUSAL-AI]` 使用了方括号，违反规范（应为 `CAUSAL-AI`） |

### 🔴 7. 测试缺失

- 无 `test/unit/causal-ai/` 目录
- `planner.ts`、`learner.ts`、`world.ts`、`command-executor.ts` 等核心逻辑均**没有任何自动化测试**
- `debug.ts` 仅为手动调试脚本

## 三、重构计划（按优先级排序）

### 阶段 1：消除重复 + 提取纯函数（低风险，高收益）

1. **提取位置/方向工具函数**
   - 新建 `src/causal-ai/core/utils/position.ts`
   - 提取 `getAgentPos(state)`、`applyDirection(pos, action)`、`isMoveAction(action)`
   - 替换 `planner.ts`、`game-controller.ts`、`world.ts` 中的重复 `switch` 和正则解析

2. **统一状态提取**
   - 在 `world.ts` 上添加 `getCurrentState(): State` 方法，封装 `stateToPredicates(...)` 的重复调用
   - 替换 `executor.ts`、`command-executor.ts`、`game-controller.ts` 中的重复代码

3. **统一动作类型**
   - 删除 `web/types.ts` 中的 `ActionType`，直接复用 `core/world/types.ts` 的 `Action`
   - 同步修改 `web/` 层所有引用

4. **修复视野不一致**
   - 确认 CLI 和 Web 的默认视野范围，将 `range` 提取为具名常量 `DEFAULT_VIEW_RANGE = 2`（5×5）或 `3`（7×7），统一两处使用

### 阶段 2：拆分 `renderer.ts`（中风险，高收益）

将 689 行的 `WorldRenderer` 拆分为：

| 新文件 | 职责 |
|---|---|
| `renderer/camera.ts` | `Camera` 类：RAF 跟踪、viewport 计算、transform 应用 |
| `renderer/player-entity.ts` | `PlayerEntity` 类：玩家 DOM 创建、方向标更新、移动动画 |
| `renderer/layers/` | `BlockLayer`、`EntityLayer`、`ItemLayer`：各层 DOM 管理 |
| `renderer/styles.ts` | 动画 `@keyframes` 的 CSS 注入（或改为外部 CSS 文件） |
| `renderer/index.ts` | `WorldRenderer` 仅负责协调各子渲染器 |

### 阶段 3：拆分 `game-controller.ts` 的规划逻辑（中风险，高收益）

1. **提取规划报告生成器**
   - 新建 `src/causal-ai/web/plan-reporter.ts`
   - 将 `planTo()` 中所有 `uiManager.addPlanLog(...)` 替换为返回 `PlanReport` 纯数据对象
   - `game-controller.ts` 只负责调用 `plan()`，拿到结果后委托 `PlanReporter` 格式化并输出

2. **封装 `plannedActions`**
   - `plannedActions` 改为 `private`
   - 提供 `setPlan(actions)`、`clearPlan()`、`shiftPlan(): Action \| null` 等显式方法
   - `command-executor.ts` 改为调用这些方法，不再直接操作数组

3. **移除字符串索引访问**
   - `main.ts` 中 `controller["render"]?.()` 改为调用 `controller.forceRender()` 等公开方法

### 阶段 4：DOM 操作收口 + UI 解耦（低风险，中收益）

1. **UIManager 扩展**
   - 添加 `clearCmdInput()`、`setViewButtonText(mode)` 等方法
   - `main.ts` 中所有直接 DOM 操作迁移到 UIManager

2. **CSS 外置**
   - `renderer.ts` 中的 `innerHTML` 内联样式改为外部 CSS 类
   - `@keyframes` 动画迁移到 `pages/causal-ai.html` 的 `<style>` 或独立 CSS 文件

### 阶段 5：添加单元测试（高收益，必需）

新建 `test/unit/causal-ai/` 目录，优先覆盖：

| 测试文件 | 测试内容 |
|---|---|
| `world.test.ts` | 移动、撞墙、拾取钥匙、开门、到达终点的状态变化 |
| `planner.test.ts` | BFS 规划基本路径、无法到达目标、深度限制 |
| `learner.test.ts` | 经验记录、规则提取、去重 |
| `command-executor.test.ts` | 各指令解析与执行结果 |
| `position-utils.test.ts` | 方向计算、位置应用 |

### 阶段 6：常量提取与配置化（低风险，低收益但规范要求）

- 新建 `src/causal-ai/core/constants.ts`
- 提取：规划最大深度、探索步数、探索延时、默认视野范围、日志标签等

---

## 四、执行顺序建议

1. **先阶段 1** — 改动面大但风险低，为后续拆分铺路
2. **再阶段 3** — 让核心逻辑可测试
3. **并行阶段 5** — 边重构边加测试，保证安全
4. **最后阶段 2** — 最复杂，放到有测试保障后再做
5. **穿插阶段 4、6** — 穿插在各阶段之间完成
