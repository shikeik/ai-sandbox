# 因果链 AI 项目 —— 从零构建可学习的智能体

## 一、项目缘起与核心目标

### 1.1 我们想做什么？
我们想做一个**纯 TypeScript (Web) 的 2D 网格游戏环境**，并在其中放置一个 AI 智能体。这个 AI **不依赖任何预定义的“如何通关”的代码**，而是通过**自由探索**自行建立对世界规则的“理解”，并能根据你下达的**高级指令**（如“抵达旗子”）自主规划并执行动作序列。

### 1.2 要解决的智能层级问题
- **第一层 (原子反射)**：看到坑就跳，看到墙就绕。这是 MLP 能做的事（写死的目标）。
- **第二层 (指令执行)**：接到“去拿钥匙”的命令，然后执行。
- **第三层 (目标导向的自主规划)**：只给最终目标“抵达旗子”。AI 自己发现门挡着，自己悟出需要钥匙，自己去找钥匙并开门。**这是我们的核心追求**。
- **第四层 (人类意图)**：你说的“通关”。

### 1.3 核心机制：因果链数据库
我们希望 AI 能够记录每一次行动的经验 `(状态, 动作, 新状态)`，并通过对比这些经验，**自动提取出通用的因果规则**。在接到新目标时，利用这些规则**反向推导**出需要执行的动作序列。

## 二、游戏世界设定 (10x10 网格)

- **玩家 (Agent)**：起始于 `(0,0)`。
- **旗子 (Goal)**：位于 `(9,0)`，抵达即胜利。
- **障碍**：一堵从 `(3,1)` 到 `(3,8)` 的垂直墙壁，只留一个缺口——**门**位于 `(3,5)`。
- **门 (Door)**：初始为关闭状态，不可通行。
- **钥匙 (Key)**：位于 `(1,8)`，站上去可拾取。
- **交互规则**：
  - 拾取：走到钥匙格 → 手中持有钥匙，钥匙消失。
  - 开门：持有钥匙时移动到门所在格 → 门打开，进入该格。
  - 通过：门打开后，该格变为空地，自由通行。

## 三、当前已实现的技术细节 (Web 原型版)

### 3.1 技术栈
- **Vite + TypeScript** (内联于 HTML)
- **Canvas 2D** 渲染
- **内存数组** 存储经验数据库 (`experienceDB`)

### 3.2 已实现的功能模块
| 模块 | 实现方式 | 代码位置/函数 |
| :--- | :--- | :--- |
| **网格渲染** | 使用 Canvas 绘制网格、墙壁、物品、玩家 | `renderWorld()` |
| **状态管理** | 扁平对象 `{ agent_x, agent_y, holds, key_exists, door_open }` | `getCurrentState()` |
| **原子动作** | `move_up/down/left/right`, `pickup`, `reset` | `applyAction()` |
| **经验记录** | 每次有效动作自动将 `{ before, action, after }` 存入数组 | `executeAction()` |
| **经验可视化** | 右侧面板逐条显示，格式化为多行易读文本 | `renderExpList()` |
| **规则泛化 (简化版)** | 检查经验库中是否存在某类成功案例，若存在则**手工生成对应的规则模板** | `generalizeExperiences()` |
| **规则库展示** | 显示规则的中文描述 | `renderRuleList()` |
| **规划与执行** | 使用 **BFS 正向搜索**，基于 `applyAction` 模拟所有可能动作，寻找从当前状态到旗子的路径 | `backwardPlan()` (实为正向) |
| **日志系统** | 操作日志和规划过程日志，支持换行显示 | `addLog()`, `addPlanLog()` |

### 3.3 当前经验数据存储格式
```typescript
interface Experience {
  before: { agent_x, agent_y, holds, key_exists, door_open };
  action: string;   // 如 "move_right"
  after:  { agent_x, agent_y, holds, key_exists, door_open };
}
```

四、当前实现的局限与“写死”之处

4.1 泛化是“假的”

· 现象：generalizeExperiences() 并没有真正对比多条经验来提取变量。它只是检查经验库中是否存在拾取成功、开门成功的记录，如果有，就硬编码生成一条规则。
· 硬编码的证据：
  · 拾取规则的条件直接使用了全局常量 KEY_POS。
  · 开门规则的条件直接使用了 DOOR_POS。
· 后果：如果你把钥匙或门换一个位置，即使你探索了新地图，泛化出的规则仍然会指向旧坐标，导致规划失败。经验库是动态的，但规则是写死的。

4.2 规划器本质是正向穷举

· 虽然按钮叫“后向规划”，但实际代码是 BFS 正向搜索。它没有利用规则进行目标回归，而是盲目尝试所有动作组合。
· 在状态空间较小时（如 10x10）可行，但不符合我们追求的“逆推因果链”的智能感。

4.3 状态表示不支持泛化

· 当前状态是固定字段的扁平对象。要表示“钥匙在某个可变位置”或“门在某个可变位置”，必须引入变量和谓词。

五、下一步：升级为真正的因果学习系统

5.1 目标

实现真正的经验泛化与后向链规划，让 AI 能够：

1. 从任意包含钥匙、门、旗子的地图布局中自行学习因果关系。
2. 规则与具体坐标解耦。
3. 使用目标回归进行推理。

5.2 需要改造的核心模块

A. 状态表示重构：从扁平对象到谓词集合

```typescript
// 旧: { agent_x: 0, agent_y: 0, holds: null, ... }
// 新: Set<string>
// 例如: new Set([ "at(agent,0,0)", "at(key,1,8)", "door_closed(3,5)", "holding(none)" ])
```

谓词格式：relation(subject, ...args)。使用字符串集合，便于匹配和合一。

B. 经验泛化算法：从具体经验中提取带变量的规则

输入：多条同一动作类型的经验（如多次在不同位置的“向右移动”）。

算法步骤：

1. 对比各经验的 before 状态，找出不变的关系（如“右边一格是空地”）和变化的部分（如具体坐标）。
2. 将变化的部分替换为变量（如 ?x, ?y）。
3. 同理分析 after 状态，提取效果谓词。
4. 输出通用规则，例如：
   ```
   动作: move(?dir)
   前提: at(agent,?x,?y) ∧ adjacent(?x,?y,?nx,?ny,?dir) ∧ cell_empty(?nx,?ny)
   效果: ¬at(agent,?x,?y) ∧ at(agent,?nx,?ny)
   ```

C. 后向链规划器

输入：当前状态谓词集、目标谓词集（如 at(agent,9,0)）、规则库。

算法：递归目标回归。

1. 如果当前状态满足目标，返回空计划。
2. 遍历规则库，找到效果谓词能与目标谓词合一的规则。
3. 将该规则的前提条件（替换绑定后的）作为新的子目标集合。
4. 递归求解子目标。
5. 返回 子计划 + 当前规则的动作。

合一 (Unification)：将目标 at(agent,9,0) 与规则效果 at(agent,?x,?y) 匹配，生成绑定 {?x: 9, ?y: 0}。

5.3 升级后的预期效果

· 你可以在任意位置放置钥匙和门，让 AI 自由探索一段时间后，它就能自动生成适用于新地图的规则。
· 给定新目标，规划器会逆向推理：“要抵达旗子，必须先通过门；要通过门，必须持有钥匙；要持有钥匙，必须先移动到钥匙位置” —— 完全自动，无需硬编码。

八、CLI 命令行版本开发（2026-04-13）

### 8.1 目标
创建一个可用于快速验证 AI 算法的命令行版本，脱离浏览器环境，便于调试和自动化测试。

### 8.2 已实现
- **5x5 局部视野**：Agent 只能看到周围 5x5 格子，模拟生物有限感知
- **纯符号表示**：使用全角字符表示世界（＠=玩家 🔑=钥匙 🚧=关闭的门 等）
- **6 个基础动作**：上/下/左/右/互/等
- **面朝方向机制**：互动时根据面朝方向判断（如面朝门才能开门）
- **钥匙-门机制**：拾取钥匙 → 面朝门互动 → 消耗钥匙开门 → 到达终点
- **地图系统**：
  - 内置 default 地图（新手教学）
  - 支持 gamedatas/maps/xxx.json 自定义地图
  - 地图数据与源码分离，gamedatas/ 目录被 git 忽略
- **状态显示**：始终显示玩家位置、面朝方向、钥匙持有状态
- **两种启动方式**：菜单选图 / 命令行参数 `--map xxx`

### 8.3 遇到的坑
1. **地图设计陷阱**：最初门旁边留空，玩家可以绕路直接到终点，无需拿钥匙开门
2. **视野大小**：3x3 视野太小，玩家看不到钥匙位置；改为 5x5 更合适
3. **指令长度**："互动"太长，改为单字"互"；"等待"改为"等"
4. **数据与代码分离**：最初地图配置写在 TS 源码里，后改为 JSON 数据文件

### 8.4 文件结构
```
gamedatas/maps/level1.json     # 用户数据（git 忽略）
src/causal-ai-cli/
├── main.ts                    # 入口（参数解析、选图菜单）
├── maps.ts                    # 内置 default 地图
├── map-loader.ts              # 地图加载器（内置 + JSON）
├── world.ts                   # 游戏世界逻辑
├── renderer.ts                # 视野渲染
└── types.ts                   # 类型定义
```
