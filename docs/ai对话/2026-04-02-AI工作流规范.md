# AI 编程助手工作流规范

> 本文件面向后续接手的 AI 编程助手。如果你刚接触本项目，**请严格遵循以下流程**，不要跳过任何步骤。

---

## 第一步：与用户确认下一步行动

**不要直接改代码。**

请先阅读本文件和 `AGENTS.md`，然后向用户汇报：
1. 你已了解项目背景和当前状态；
2. 请用户明确告诉你**下一步要做什么**（新功能 / 继续重构 / Bug 修复 / 其他）。

在用户确认之前，只进行**只读调研**，不写代码。

---

## 第二步：全面评估代码质量

一旦用户确认要修改代码，先进行全面质量评估：

1. 读取 `.kimi/skills/code-quality-refactoring/SKILL.md`；
2. 通读 `src/` 下所有核心文件（尤其是 `main.js`、各模块入口）；
3. 按以下维度识别问题：
   - **重复代码**（DRY）
   - **过长函数/过大类**（SRP）
   - **魔法数字/硬编码字符串**
   - **发散式变化**（改一个需求要动多处）
4. 将发现的问题按**优先级**（高/中/低）整理成清单，向用户汇报。

---

## 第三步：规划重构/开发步骤

如果用户同意继续，请按以下原则规划：

1. **小步快跑**：每次只改一个独立的点；
2. **从低风险、高价值项开始**（如提取类、合并重复方法）；
3. **Bug 修复优先于重构**；
4. **每次修改前在代码中预留/补充调试日志**，确保修改后可验证；
5. 列出 Todo 清单，逐步执行。

---

## 第四步：隔离开发

1. 检查当前分支是否有未提交更改；
2. 若有，先提交到当前分支；
3. 切到 `main` 分支并合并当前开发分支；
4. 基于最新 `main` 创建新的隔离分支（如 `refactor/xxx` 或 `feature/xxx`）；
5. 所有代码修改在新分支上进行。

---

## 第五步：执行与验证

每完成一个子任务：

1. **运行 `npx eslint src/`**，确保零错误；
2. **运行 `npm run build`**，确保构建通过；
3. 如果涉及交互逻辑，**补充 `[TAG]` 格式的控制台日志**（如 `[AI]`, `[RENDER]`, `[GAME]`），方便用户在浏览器中验证；
4. 提交代码，commit message 使用**全中文**。

---

## 第六步：用户验收

每次提交后，向用户提供：

1. **本次修改的摘要**；
2. **详细的验收操作步骤**（按模块/功能列出）；
3. **预期应看到的日志输出**（带 `[TAG]` 的示例）；
4. 让用户运行 `npm run dev` 进行手动验证，收集用户反馈或日志文件。

如果验收未通过，分析日志，修复问题，重新提交。

---

## 第七步：合并与交接

当一轮任务全部完成并验收通过后：

1. 将隔离分支 **合并回 `main`**；
2. 在 `docs/ai对话/` 下新建以日期命名的交接/总结文档；
3. 回到**第一步**，向用户确认**下一步行动**。

---

## 当前项目关键信息（速查）

- **分支**：`main` 为默认分支，重构成果均已合并。
- **技术栈**：原生 JS (ESM) + Vite + Canvas 2D + 纯 CSS。
- **代码风格**：Tab 缩进、无分号、禁止 `var`、注释用中文。
- **构建命令**：`npm run build`
- **Lint 命令**：`npx eslint src/`
- **启动命令**：`npm run dev`（端口 4000）

---

## ⚠️ 禁止事项

- 不要一次性重构整个代码库；
- 不要在没有日志的情况下交付交互逻辑改动；
- 不要在用户未确认下一步时擅自开始新任务；
- 不要删除任何 `.kimi/skills/` 中的技能文件，除非用户明确要求。
